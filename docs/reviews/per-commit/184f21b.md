## Track 4 — Per-commit Doodlestein

**Strategic**: The `audit_returned_facts.fact_id` foreign key references `memory_entries(id)` but `memory_entries.id` is a rowid INTEGER — if a memory entry is deleted, the audit row becomes a dangling FK with no cascade rule specified; adding `ON DELETE CASCADE` or `ON DELETE SET NULL` (depending on whether orphaned audit rows are acceptable) would make the retention semantics explicit before the table has any data.

**Adversarial**: Hunk 5 changes the hardcoded `5` in `test_migration_v4_to_v5_happy_path` to `SCHEMA_VERSION` (now 6), which breaks the test's intent — that test is validating the v4→v5 migration path and should assert `user_version = 5` after it runs, not the head version; using the constant here is the opposite of the M7 literal-pinning rationale applied in Hunk 2.

**Regret**: The `idx_memory_entries_valid_until_id` partial index on `memory_entries` is added inside the v6 migration block but is unrelated to the audit tables — it's a speculative optimization for an existing table, and if it causes query-planner surprises or is superseded by a future index strategy discussion, it will be the first thing removed from this migration.

- Strategic (FK cascade): WAIVE — discussion 029 YAGNI 1 settled "no explicit ON DELETE; PRAGMA foreign_keys stays OFF". Cascade rules are moot under unenforced FKs. Codex track 2 separately flagged the implicit-OFF assumption — captured as BL-015.

Pattern matches existing `test_migration_from_v3_preserves_data` (line ~643). The M7 literal-pin principle applies to migration-block writes, not to head-version test assertions.
