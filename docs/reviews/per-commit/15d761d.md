## Track 4 — Per-commit Doodlestein

**Strategic**: Move `audit_start` in `mcp_tools.rs` to after the input-validation and project-id resolution blocks — currently it starts before `pid` is resolved and before any early-return guards, so `took_ms` will include validation overhead and never match what the DB function receives as `scope`/`project_id`; the timer should start immediately before the `memory_search` call to measure only search + filter latency.

**Adversarial**: The `mcp_tools::search` async fn acquires a `std::sync::Mutex` (inside `record_search_audit_best_effort`) synchronously — `.lock().unwrap()` blocks the Tokio thread. With concurrent search requests this can deadlock or starve the executor. The best-effort wrapper silences errors but does not protect against blocking; the fix is `tokio::task::spawn_blocking` wrapping the audit write, or using `tokio::sync::Mutex` in the DB layer.

**Regret**: The `cli.rs` audit call — `scope` in the CLI path is `Option<&str>` derived from a different code path than the MCP path's `project_id: &str`; when the wrapper signature is used in both callsites the `scope` semantics diverge (CLI passes git-inferred optional scope, MCP passes resolved project_id string), making unified audit-table queries ambiguous. This inconsistency is likely to surface during the audit-stats or doctor command work and force a callsite rework.

## Lead disposition

- **Strategic: REJECT** (stale design decision). Discussion 029 Topic 1 explicitly chose Option B (timer starts AFTER input validation, BEFORE embedding) over Option A (timer immediately before `memory_search`). Rationale: embedding generation is part of caller-perceived search latency; `took_ms` should reflect what the user waited for, not just the SQLite query time. Doodlestein's suggestion would re-implement the rejected Option A.
- **Adversarial: WAIVE** (project-wide pattern). The audit hook's std::sync::Mutex acquisition matches the precedent set by `record_recall`, `insert_synthesis_with_links`, `increment_metric`, and every other Db method in the project. Discussion 029 R3 explicitly accepts this: "Sync (~100-400µs); matches `record_recall` precedent". Switching to tokio::sync::Mutex or wrapping every DB call in spawn_blocking is a project-wide refactor outside F-002 scope. Codex Track 2 (this same review) independently confirms: "std::sync::Mutex safe in async context — no `.await` happens while locked; critical section is short (~100-400µs). No runtime starvation risk at this scale. Revisit only if writes spike into multi-ms territory or contention patterns change." If this concern recurs under observed load, file a targeted BL.
- **Regret: REJECT** (misread the diff). Both call sites pass `Option<&str>` with identical semantics: `None` means "global / cross-project", `Some(pid)` means "project-scoped". CLI: `scope: None` when `--global` flag is set, `Some(&project_id)` otherwise. MCP: `project_id: None` when `params.scope == Some("global")`, `Some(pid)` otherwise. The audit-table column receives the same shape from both sources; BL-014 audit-stats SQL will treat them uniformly. No divergence exists.
