## Track 4 — Per-commit Doodlestein
**Strategic**: The `JULIANDAY(me.valid_until) - JULIANDAY(a.searched_at) <= 7` threshold in both SQL constants is a magic number with no named constant or comment explaining why 7 days was chosen; extracting it as a named const (e.g., `SUPERSESSION_WINDOW_DAYS`) would make future tuning deliberate rather than a grep-and-pray across two SQL strings.
**Adversarial**: `F002_SUPERSESSION_SQL` is marked `#[allow(dead_code)]` and returns a windowed GROUP BY / HAVING result, while `AUDIT_STATS_SUPERSESSION_COUNT_SQL` is the one actually called — the two queries answer subtly different questions (time-bucketed spike detection vs. flat 30-day count), and no caller currently uses the former; if the intent was to power a future alerting path, the semantic divergence will silently produce wrong alert logic when someone eventually wires it up.
**Regret**: The `#[allow(dead_code)]` on `F002_SUPERSESSION_SQL` is the most likely rework target — either it gets deleted when the dead-code warning becomes noise during a clippy pass, or the allow-attribute gets removed and the query gets properly wired (or the query is redesigned), meaning the constant as written is unlikely to survive v0.0.1 in its current form.
