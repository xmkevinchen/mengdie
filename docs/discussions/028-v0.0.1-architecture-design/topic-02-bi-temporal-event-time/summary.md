# Topic: Bi-temporal event_time vs ingested_at column inclusion

## Current Status
**Converged** (Round 2). REJECT permanently. Alternative: optional `valid_from` parameter on `memory_ingest`.

## Round History
| Round | Score | Key Outcome |
|-------|-------|-------------|
| 1 | exploratory | 3-of-5 reject; codex DEFER; gemini agnostic |
| 2 | converged | 4-of-5 reject (challenger + gemini updated); codex dissent recorded |

## Context

The proposed v0.0.1 architecture borrows Graphiti's bi-temporal
model: facts have `event_time` (when fact was true) separate from
`ingested_at` (when mengdie recorded it). architecture-reviewer
accepts this as correctly placed in Storage layer.

challenger argues the borrow may not apply to mengdie's workflow:
in Graphiti's chat use case, the two timestamps genuinely differ
(user says "we decided X last Tuesday"; system processes today). For
AE pipeline artifacts, the artifact is generated by a completed
pipeline run and ingested within seconds — `event_time ≈ ingested_at`
for the modeled flow.

challenger's falsifiable demand: produce one AE artifact in
production where `event_time` differs from `ingested_at` by more than
60 seconds. If none exists in v0.x's production corpus (~214
facts), the column is borrowed schema with no observable payoff.

The exception scenario challenger named: post-hoc documentation
(operator writes `conclusion.md` today about a decision made three
months ago). Whether this is a first-class workflow or rare edge
case is operator domain knowledge.

## Constraints

- Blueprint §6 implementation principle: do not borrow patterns that
  serve another use case if the payoff for mengdie's actual workflow
  is not demonstrable.
- Schema migration cost is non-trivial — adding a column requires
  migration logic even at small scale.
- Future post-hoc documentation workflows are not currently in v0.0.1
  scope.

## Key Questions

- In the operator's actual AE workflow over the past 6 months, are
  there concrete instances where the artifact creation time and the
  underlying decision time differed by more than ~60 seconds?
- Is post-hoc documentation a workflow the operator intends to
  support in v0.0.1?
- If both answers are no, what is the trigger condition for adding
  the column later (e.g., "first ingestion of a retrospectively-dated
  artifact")?
