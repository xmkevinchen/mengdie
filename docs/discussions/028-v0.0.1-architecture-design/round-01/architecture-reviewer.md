# Round 1 — Architecture Reviewer

## Findings (with file:line evidence)

### Topic 1 — Storage abstraction timing + mechanism

**The search-split refactor is NOT in v0.0.1 scope as a standalone item,
but the two-ingest-paths defect fix induces it as a side effect.**

Evidence:

- `src/core/search.rs:80-206`: `impl Db { fn memory_search() }` —
  search functions are extension methods on `Db`. The caller pattern is
  `self.db.memory_search(...)` (mcp_tools.rs:211). This conflates the
  Storage layer with retrieval logic.
- `src/core/mcp_tools.rs:209-244`: the `search` tool calls
  `self.db.memory_search(...)` and `self.db.search_fts(...)` directly on
  `Db`. No module-level `search::` call site exists.
- `src/core/mcp_tools.rs:306-395`: the `ingest` tool constructs
  `super::db::NewMemory` inline, calls `self.db.insert_memory()` and
  `self.db.insert_memory_resolving()` directly, embeds inline, and runs
  the contradiction check inline — bypassing `ingest::ingest_document`
  entirely. This is the two-ingest-paths defect.

The two-ingest-paths defect fix requires consolidating `mcp_tools.rs`
ingest through `ingest::ingest_document`. That consolidation requires
`ingest::ingest_document` to accept the same parameters that
`mcp_tools.rs` currently constructs inline. Once the ingest path
consolidation is done, the question of whether `search.rs` functions stay
as `impl Db` or move to module-level is independent.

**The search-split refactor does NOT fall out automatically from the defect
fix.** The defect is on the ingest side (`mcp_tools.rs:282-395`); search
(`mcp_tools.rs:175-276`) continues to call `self.db.memory_search()`
regardless.

**Mechanism verdict: free functions over a `Connection` handle, not a
Rust trait, and not yet.**

Reasoning:

1. The YAGNI rule from the analyze phase holds: `Storage` trait earns
   its keep when Kuzu (Tier 2) is in the same sprint. Kuzu has no commit
   date.
2. Moving `search_fts` and `memory_search` from `impl Db` to free
   functions (`search::memory_search(db: &Db, ...)`) costs nothing at the
   call site — callers just use `search::memory_search(&self.db, ...)`
   instead of `self.db.memory_search(...)`. The boundary becomes real
   without a trait.
3. A trait at this point adds a generic parameter or dyn dispatch to
   every call site for zero benefit (one impl, no mock needed — tests use
   `Db::open_in_memory()`).
4. The free-function refactor is the correct intermediate step: it makes
   Retrieval a real module boundary, satisfies the analyze-phase
   conditional for `Storage` trait acceptance, and costs one PR worth of
   caller updates.

**Decision: include the search-split (search.rs → free fns over `&Db`)
as part of the two-ingest-paths defect fix PR, since both touch
`mcp_tools.rs` callers. Do NOT introduce a `Storage` trait in v0.0.1.
Defer the trait to the Tier 2 trigger (Kuzu sprint).**

Trigger for Storage trait: "Kuzu integration sprint begins, requiring a
common CRUD interface over both SQLite and Kuzu."

### Topic 2 — Bi-temporal `event_time`

**Reject the column for v0.0.1. The challenger's falsifiable demand holds.**

Evidence from v0.8.0 schema: the existing `memory_entries` table has
`valid_from` (ingest timestamp), `valid_until` (supersession timestamp),
`superseded_by`. There is no `event_time` column. The current schema has
been in production across 214 facts.

Workflow analysis: AE pipeline artifacts are generated by a completed
pipeline run (the agent writes `conclusion.md`, the watcher or MCP push
fires). The delta between "decision made" and "artifact written" is the
time for the agent to finish writing the file — measured in seconds, not
days. The agent does not write retroactively dated documents as a normal
workflow; it writes what it did right now.

The only scenario where `event_time` ≠ `ingested_at` by > 60 seconds:

1. **Post-hoc documentation** — operator writes today about a decision
   made three months ago. This is NOT a pipeline artifact; it would be a
   manually authored note. Blueprint §9 locks out generic markdown
   ingestion. So this scenario is out of scope by identity.
2. **Slow pipeline** — the agent takes > 60 seconds to write the artifact
   after the decision is made. Pathological; not a design requirement.
3. **Replay import** — bulk import of historical AE discussions. This
   is a one-time migration operation, not a runtime workflow. It could be
   handled by accepting a caller-provided `valid_from` override on the
   ingest MCP tool, which costs one optional parameter — not a schema
   column.

Graphiti's `event_time` is justified because its facts are extracted from
chat where "we decided X last Tuesday" is a first-class input. mengdie's
facts are pipeline outputs stamped at write time. The patterns diverge.

**Decision: reject `event_time` column permanently for v0.0.1. File a
note in the blueprint under §8 open questions: if bulk import of
historically-dated artifacts becomes a workflow (distinct from runtime
ingest), accept an optional `valid_from` override on `memory_ingest`
instead of a schema column. That optional parameter costs nothing and
covers the actual use case.**

This is not "defer with trigger" — it is a permanent rejection of the
bi-temporal column for mengdie's workflow, with an alternative path for
the only legitimate use case (caller-supplied `valid_from`).

### Topic 3 — Reflection module consolidation + Reflector trait

**Defer consolidation. Reject Reflector trait for v0.0.1 even if sqlite-vec
spike succeeds.**

Evidence:

- `src/core/dreaming.rs:1-9`: imports `clustering::cluster_memories`,
  `synthesis::{build_synthesis_prompt, parse_synthesis_response, ...}`.
  These are the only callers of clustering.rs and synthesis.rs.
- `src/core/clustering.rs:1-38`: pure algorithm module. No other module
  imports it except dreaming.
- `src/core/synthesis.rs` (file header): pure prompt builder + JSON
  parser. No other module imports it except dreaming.

The module boundary between clustering, synthesis, and dreaming is
currently fictional as an API boundary — it is a code organization
boundary only. Consolidation into one `dreaming.rs` would reduce file
count without changing any external interface.

**BUT: the sqlite-vec spike may render the question moot.**

If the sqlite-vec compatibility spike succeeds (blueprint §10 item 1),
`clustering.rs` becomes a candidate for deletion (025 verdict:
CONDITIONAL-DELETE if ANN replaces hand-rolled cosine clustering). In
that case:
- `clustering.rs` is deleted, not merged
- `synthesis.rs` remains (prompt building is orthogonal to how clusters
  are found)
- `dreaming.rs` shrinks; it now calls sqlite-vec ANN for neighbors and
  `synthesis.rs` for the LLM pass

Forcing the merge now, before the spike outcome is known, means:
- If sqlite-vec succeeds: undo the merge (clustering disappears, synthesis
  stays separate)
- If sqlite-vec defers: the merged module is now a god module

Neither outcome justifies paying the merge cost pre-spike. Deferring costs
nothing.

**On the Reflector trait:** the analyze-phase "defer with trigger" verdict
specified "≥2 reflection strategies." Does sqlite-vec adoption create a 2nd
strategy?

No. sqlite-vec ANN is a replacement for clustering's neighbor-finding
algorithm within the same strategy (cluster → synthesize → store). It does
not introduce a second reflection strategy — it changes the implementation
of step 1 inside the existing strategy. Two reflection strategies would
mean: (A) the existing dream pass, and (B) a fundamentally different
reflection approach (e.g., entropy-triggered summary, or a Reflexion-style
critique loop). sqlite-vec changes how clusters are found, not what
reflection means.

**Therefore the Reflector trait trigger does NOT fire from sqlite-vec
adoption alone.**

**Decision:**
- Defer module consolidation until sqlite-vec spike completes. If ANN
  replaces clustering: clustering.rs is deleted, synthesis.rs stays
  separate. If ANN defers: revisit in v0.0.2 sprint scoping.
- Reject Reflector trait for v0.0.1. Trigger remains: "≥2 fundamentally
  different reflection strategies exist in the same sprint." sqlite-vec
  adoption does not meet this condition.

### Topic 4 — A-MEM bidirectional update deferral trigger

**Composite trigger: corpus size AND measured retrieval quality signal.**

The analyze phase listed three candidate triggers: corpus size, retrieval
quality degradation, external replication. My position:

**Reject corpus size alone.** 1k facts is arbitrary. A 500-fact corpus
with high entity overlap and stale facts may degrade faster than a 2k-fact
corpus with clean partitions. Size is a proxy for complexity, not a
direct measure of the problem A-MEM solves.

**Reject external replication alone.** A-MEM replication in 2026 would
tell us the technique works in principle, not that mengdie's specific
workflow needs it. The operator's loop is AE-artifact-driven, not
conversation-derived — the workload profile differs from A-MEM's test
cases.

**Accept composite trigger with a primary and a qualifying condition:**

Primary condition: **retrieval quality measurably degrades** — defined as:
avg `score` of top-3 results in the domain audit log drops below 0.35
(the score is 0–1 normalized RRF; 0.35 is approximately the 50th
percentile for a dual-ranker hit) over a rolling 30-day window of at
least 20 queries.

Qualifying condition: **corpus size > 500 facts** AND **average entity
cluster size > 5** (meaning entities are being reused across many facts,
which is the structural precondition that makes bidirectional update
useful — if every fact has unique entities, no cluster re-evaluation is
needed).

Rationale: the domain audit table (blueprint §5 P0 instrumentation) logs
every search call with what was returned. This is the instrumentation
v0.0.1 will have. The trigger is measurable from that table without
additional infrastructure.

**Decision: A-MEM is deferred from v0.0.1. Trigger: avg top-3 search
score < 0.35 over 30-day rolling window of ≥ 20 queries, AND corpus >
500 facts, AND avg entity cluster size > 5. All three conditions must
hold simultaneously.**

## Open Questions

1. **Does the two-ingest-paths defect fix require touching `search.rs`
   callers?** My analysis says no — the defect is on the ingest side only.
   But if the fix refactors how `mcp_tools.rs` accesses `db`, it may be
   natural to move `search_fts` / `memory_search` to free functions in the
   same PR. This is a scope question for the defect-fix BL.

2. **Should the search-split be a prerequisite for the defect fix, or a
   follow-on?** If the defect fix creates a `pub fn ingest_document()` in
   `ingest.rs` that `mcp_tools.rs` calls, the search side stays as `impl
   Db` calls. A follow-on PR can move search to free functions. No
   functional coupling between the two.

3. **What is the caller-supplied `valid_from` interface?** If we accept
   an optional `valid_from` parameter on `memory_ingest` for bulk import,
   it belongs in `IngestParams` in `mcp_tools.rs:79-99`. Needs a framing
   decision before the ingest defect-fix BL is filed.
