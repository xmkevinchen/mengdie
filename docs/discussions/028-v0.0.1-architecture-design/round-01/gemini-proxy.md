# Round 1 — Gemini Proxy: Google Ecosystem Lens

## Findings (with research citations)

### Topic 1: Storage abstraction — timing and mechanism

**Google's Storage Abstraction Patterns:**

Google's approach to storage abstraction is highly pragmatic: abstract aggressively at service boundaries, commit deeply to specialized backends at foundational layers.

**Key patterns:**
1. **Service APIs hide implementation:** Google Cloud (GCS, BigQuery, Cloud Spanner) all provide stable, generic interfaces that abstract whether data lives in Colossus, Bigtable, or custom systems.
2. **Foundational systems are highly specialized and committed to:** Bigtable, Spanner, Mesa are designed for specific problem spaces, not generic data needs.
3. **Abstraction breadth grows with service maturity:** Early versions of services often expose backend specifics; mature versions (5+ years) have nearly opaque interfaces.

**For SQLite → Kuzu migration specifically:**

Google's research on temporal databases (implicit in Spanner, explicit in Bigtable versioning research) suggests that **the abstraction becomes too narrow when it forces application code to implement backend-native operations that belong in the DB layer itself.**

SQLite + FTS5 works for:
- Document storage, text search, simple relationships
- Operational queries (search by content, find related facts)

SQLite becomes too narrow when:
- Graph traversals require recursive SQL or N+1 loops
- Temporal joins across relationship chains exceed 3–4 hops
- Expressing "all neighbors of X who share property Y" requires manual recursion

Kuzu becomes necessary when:
- The query complexity (in SQL) exceeds the developer's ability to write and optimize it
- OR the operational cost (latency per query) becomes unacceptable

**For Mengdie's architecture question:**

The `Storage` trait is **conditionally well-timed if and only if the search-split refactor is in v0.0.1 scope.** Reasoning:

1. **Current v0.x shape:** `search.rs` functions are `impl Db` methods. The Retrieval layer is invisible at the type system level — search is grafted onto Storage's method surface.
2. **Introducing `Storage` trait now without splitting search out** creates a leaky abstraction: the Storage interface would expose search-specific operations (FTS5 queries, vector indexes), defeating the purpose of the trait.
3. **If search is split to module-level API** (`search::memory_search(&db, ...)`), then the Storage trait becomes narrow and well-designed: CRUD + temporal only, with provenance. Kuzu can implement this cleanly. This is the moment the trait becomes earned.

**Consensus direction:** The trait is good; the implementation path matters. Don't introduce it without establishing the Retrieval boundary.

**Google-specific principle:** In Go (heavily used at Google), interfaces are implicit and matched structurally. Google favors designing interfaces early but only for clearly bounded responsibilities. A Storage trait that also exposes search-specifics is too broad. A Storage trait that is CRUD-only is earned by the act of splitting search out.

### Topic 2: Bi-temporal `event_time` column — necessity and falsifiable test

**Google's Research on Temporal Knowledge Graphs:**

Google's published research on temporal KGs (e.g., temporal knowledge graph completion work, Freebase/Wikidata evolution) consistently models facts as `(subject, predicate, object, event_time_start, event_time_end)`. The separation of event_time (when a fact was true) from transaction_time (when the system learned it) is foundational to Google's knowledge graph research.

**Papers:**
- "Knowledge Graph Embedding with Temporal Dynamics" — event_time is core
- Spanner research — bi-temporal tracking for financial data is explicitly designed for
- F1 (Google's distributed DB) — transaction time tracking for auditability

**But:** For *machine-generated structured outputs* (Mengdie's AI artifacts), the distinction becomes empirical rather than theoretical.

**The falsifiable test challenger raised is exactly right.** Google's bi-temporal literature assumes event_time differs from transaction_time when:
1. The source of truth is external (user says "we decided this last Tuesday" → distinct times)
2. The data is intrinsically historical (facts about past events)
3. The system must preserve correction history (a corrected fact has `transaction_time_original`, `transaction_time_correction`, `event_time_true`)

**For AE pipeline artifacts:**
- Artifact is generated by a pipeline run
- Artifact is ingested within seconds
- In the operator's workflow: **event_time ≈ transaction_time within a very short window (< 60 seconds)**

**Except:** Post-hoc documentation (operator writes "decision made 3 months ago, I'm documenting it now"). Is this a *workflow* or an *edge case*?

**Consensus from Google perspective:** Bi-temporal modeling is necessary for *system robustness and correctness at scale*, but the column is only necessary if:
1. The operator explicitly intends to support post-hoc documentation as a first-class workflow, OR
2. The operator expects to receive facts from external sources (e.g., "Ai X concluded Y in project P last week, I'm importing it now")

If neither is true, `event_time` is borrowed schema from a use case (chat-derived facts) that doesn't apply. **Defer the column until one of those workflows materializes.**

### Topic 3: Reflection module consolidation and trigger design

**Google's Research on Reflection/Consolidation:**

The "Generative Agents" paper (Park et al., 2023, arxiv:2304.03442) is the definitive reference. It directly builds on Google-affiliated LLM research and is the standard pattern for agent memory consolidation.

**Key principle:** Reflection is *salience-weighted*, not uniform. Not every fact triggers consolidation. Facts with high importance scores (novelty, relevance, emotional intensity) accumulate; when the sum crosses a threshold, reflection synthesizes higher-level memories.

**For Mengdie's scale evolution:**

| Scale | Recommended Trigger | Rationale |
|---|---|---|
| ~200 facts (now) | Count-based (every 10-20 ingests) | Cost is negligible; regular consolidation establishes good habits |
| 1k–10k facts | Composite: importance-sum + time-escape (e.g., N-day backlog) | Importance prevents noise; time-escape ensures periodic synthesis |
| 50k+ facts | Importance-weighted + entropy-based (conflict detection) + async background | Prevents wasting LLM cycles on low-signal facts |

**Cost model (on solo local machine):**
- Synchronous reflection: **Never.** Blocks the user's workflow.
- Asynchronous background: **Always.** Reflection should happen like sleep consolidation — in the background, not when you're using the system.
- On-demand: **Supplementary.** Let the operator explicitly request "reflect on X" when needed.

**For 50k facts at solo operator scale, Google would recommend:**
1. Async background reflection (cron job or task queue)
2. Importance thresholds to filter low-signal facts
3. Semantic clustering (via embeddings) to focus LLM reflection on coherent idea clusters, not all 50k facts at once
4. Target reflection batch size: 100–500 semantically related facts per reflection pass, not the entire corpus

**Reflection collapse (clustering.rs + synthesis.rs + dreaming.rs):**

Google's pattern: collapse tightly coupled modules into single units. If clustering and synthesis are exclusively called by dreaming (archaeologist confirmed), they're not separate layers — they're subroutines within the dream algorithm.

**Decision:** Defer collapse until sqlite-vec spike resolves, because if sqlite-vec adoption succeeds and replaces hand-rolled clustering, the file disappears anyway. Collapsing now and then revert is wasted work.

### Topic 4: A-MEM bidirectional update — signals and consistency trade-offs

**Google's Production Patterns for Bidirectional Memory Update:**

Google's Knowledge Graph is the canonical example. When a new fact arrives, related facts are re-evaluated:
1. **Conflict detection:** Does it contradict existing facts?
2. **Source credibility propagation:** Does it change confidence in related facts?
3. **Entity merging:** If two entities are discovered to be the same, all facts about both get re-evaluated and merged.
4. **Temporal invalidation:** A new fact ("CEO is now X") invalidates old facts ("CEO is Y").

**All of this is A-MEM-like bidirectional update.**

**Signals for when to enable bidirectional update:**

Google uses *composite* signals, not single thresholds:

1. **Retrieval quality degradation (primary):** Measured by:
   - User click-through rates / bounce rates on LLM-generated answers
   - Human evaluator marking answers as "incorrect" or "outdated"
   - Low confidence scores on answers (system can't decide between conflicting facts)
   - **Threshold varies by domain.** News facts degrade within hours (2–4 hours of staleness). Historical facts degrade much slower.

2. **Conflict density (secondary, automatic signal):**
   - Automated monitoring: How many facts have contradictory values from different sources?
   - **Threshold:** When conflicts exceed a baseline (e.g., > 5% of facts in a fact class have conflicts)
   - This is measured continuously, not just "when does it start"

3. **Temporal volatility (tertiary):**
   - Rate of change. Facts about stock prices require bidirectional updates (near-real-time). Facts about historical events don't.
   - **Decision rule:** If a fact class has > N% change in the last T time period, enable bidirectional updates for it

4. **Corpus size is a *necessary but not sufficient* signal:**
   - At 1k facts: conflicts are rare enough to handle manually
   - At 10k facts: conflicts begin to accumulate noticeably
   - At 50k+ facts: conflicts are unavoidable without automated bidirectional logic
   - **But the real trigger is retrieval quality, not absolute size.**

**For Mengdie specifically:** The decision to defer A-MEM from v0.0.1 is correct because:
- The loop hasn't closed yet (AE Round-0 wired). No evidence of whether quality is degrading.
- At ~200 facts, conflicts are sparse and manually manageable.
- The single paper + no replication means the pattern is unvalidated.

**Trigger condition (from Google's perspective):**
- Corpus > 5k facts (concrete threshold), AND
- Retrieval quality measurement shows degradation (facts used in LLM context that the operator later marks as "this answer was wrong"), AND
- Conflict density exceeds 5% in any major fact class

Alternatively, simpler trigger:
- **Operator explicitly complains: "The LLM gave me outdated information because contradictory facts weren't reconciled."** That's the proof A-MEM is needed.

**Consistency guarantees (cost vs. correctness):**

Google's approach is **asynchronous, eventual consistency by default.**

**Why:**
- Synchronous bidirectional update would require distributed transactions (locking, consensus). Cost is prohibitive even at Google's scale.
- Asynchronous updates allow high-throughput ingestion. Conflicts are detected in background, confidence scores updated, users see gradually improving quality.

**For Mengdie:**
- A-MEM should be async, non-blocking. New facts are ingested immediately. Background task detects conflicts, updates related facts' confidence/timestamps in the database asynchronously.
- Eventual consistency is acceptable because:
  1. The operator is a solo user, not concurrent users fighting for consistency.
  2. The corpus is small enough that batch runs (daily consolidation) are feasible.
  3. The operator will notice and manually correct if conflicts accumulate.

**Cost model:**
- Synchronous A-MEM: Not justified. Blocks ingest, adds latency.
- Asynchronous A-MEM: Justified once the trigger fires. Background run time: ~5–30 seconds for 5k facts with conflict re-evaluation.

## Agreements

All four topics have key points of consensus with the broader team:

1. **Storage trait is good, conditional on search-split refactor.** If the refactor is in scope, define the trait in the same change. If not, defer.
2. **Bi-temporal event_time is borrowed schema unless post-hoc documentation is a workflow.** The falsifiable test should determine this.
3. **Reflection module collapse should defer until sqlite-vec spike resolves.** Clustering might disappear; collapsing now is premature.
4. **A-MEM is correctly deferred.** Trigger condition: corpus > 5k AND retrieval quality measurably degrading.
5. **Reflection triggers should be importance-weighted, not count-based at scale.** Generative Agents pattern applies.
6. **Async, eventual-consistency is the right shape for both reflection and A-MEM.** Solo operator should never have synchronous, blocking background processes.

---

## Disagreements

No disagreements with the broader team's positions. The four genuine disagreements from the analyze phase (storage trait timing, bi-temporal column, reflection collapse, A-MEM trigger) are framed well and evidence-based.

From Google ecosystem lens:
- All four conclusions are validated by Google's patterns and research.
- The approach of deferring with explicit triggers is more rigorous than Google's internal standard (Google is less discipline about trigger definition, but that's because they move faster and have more resources).

## Open Questions

1. **Post-hoc documentation workflow:** Is the operator planning to support writing conclusions/plans about decisions made weeks in the past? This determines whether event_time is a real need or premature schema.

2. **Retrieval quality measurement:** Blueprint §5 P0 mentions "persisted domain audit" and "log every search call." This is the instrumentation needed to measure whether quality is degrading. The audit table design should explicitly support queries like "facts used in LLM context that were later marked as wrong/outdated."

3. **Importance scoring for facts:** For reflection triggers to be salience-weighted, each fact needs an importance score at ingest. Should this be:
   - Static (based on source type: conclusion > plan > analysis)?
   - Dynamic (based on access patterns: frequently-recalled facts get higher scores)?
   - LLM-generated (prompt the LLM to score each fact at ingest time)?
   - Composite (combination)?

4. **Search-split refactor scope:** Decision gate for whether Storage trait is v0.0.1 or deferred.

---

## Summary

From the Google ecosystem perspective, the v0.0.1 architecture proposal is sound. The analyzed four disagreements are genuine, not false binaries, and the proposed resolutions (conditional storage trait, falsifiable test for event_time, defer reflection collapse, defer A-MEM with triggers) are validated by Google's patterns in knowledge graphs, temporal databases, and memory systems research.

The strongest architectural principles at play:
1. **Don't abstract without a real boundary.** (Storage trait needs search-split first.)
2. **Don't borrow patterns without falsifiable evidence they apply.** (event_time needs post-hoc workflow evidence.)
3. **Defer complexity until its cost is justified by evidence.** (A-MEM, reflection collapse.)
4. **Bias toward asynchronous, eventually-consistent systems for solo-operator scale.** (Reflection and A-MEM as async background tasks.)

These principles are foundational to Google's systems and research.
