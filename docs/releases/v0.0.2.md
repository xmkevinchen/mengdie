## Highlights

### Entity graph foundation (F-007)

Pre-v0.0.2: `memory_entries.entities` was a comma-separated text column. "Find all facts tagged `auth`" required `WHERE entities LIKE '%auth%'` — full-table scan with false-positive risk (`auth` matches `oauth` etc.).

Post-v0.0.2: normalized `entities` table (UUID id, project-scoped UNIQUE(project_id, name)) + `fact_entity` many-to-many with two indexes. The new `memory_entity_facts` MCP tool returns all facts tagged with a given entity name in sub-millisecond index lookup, ordered by `recall_count` desc. `contradiction.rs` candidate-finding now uses the same index — eliminates the LIKE-scan path entirely.

Backfill is one-shot, idempotent, runs inside the same transaction as the DDL. ON CONFLICT DO NOTHING on both upserts handles partial-recovery scenarios.

### Memory Lint completes the LLMWiki trio (F-008)

Karpathy's LLMWiki pattern is **Ingest / Query / Lint**. Pre-v0.0.2 mengdie shipped Ingest (`memory_ingest`) + Query (`memory_search`); Lint was a known gap. F-008 closes it with three pure-SQL checks that surface silent failure modes:

- **Orphan GC**: `PRAGMA foreign_keys = OFF` means dangling references in `superseded_by` / `memory_synthesis_links` / `audit_returned_facts` accumulate on rename / delete.
- **Unresolved contradictions**: catches half-resolved supersession state (one of `valid_until` / `superseded_by` set but not both), size-2 cycles, and ≥0.7 Jaccard entity-overlap pairs that ingest-time `contradiction.rs` missed.
- **Embedding drift**: missing embedding on non-synthesis rows, dim mismatches, and BL-022's surface (synthesis rows shipped with `embedding=NULL` in v0.0.1).

Read-only, idempotent (same DB → same findings except `generated_at`). Wire format includes an `error: Option<String>` field so callers can distinguish "DB clean" from "lint pass crashed midway" — surfaced during /ae:review and fixed pre-ship.

### MCP surface quadrupled (F-009 → F-011)

Pre-v0.0.2: 3 MCP tools (`memory_search`, `memory_ingest`, `memory_invalidate`).
Post-v0.0.2: 7 MCP tools — added `memory_get`, `memory_status`, `memory_entity_facts`, `memory_lint`.

Common pattern across the 4 new tools:

- Full UUID OR 8+ char prefix input (per F-009's `find_by_id_prefix` primitive)
- `project_id` + `scope` parameters (default scope is current project; `scope: "global"` for cross-project)
- Structured `error: Option<String>` field on outputs for failure-mode disambiguation
- Read-only tools (`status`, `lint`) explicitly don't bump recall counters

## Schema migration

`user_version 7 → 8`:

```sql
CREATE TABLE entities (
    id           TEXT PRIMARY KEY,
    name         TEXT NOT NULL,
    project_id   TEXT NOT NULL,
    first_seen   TEXT NOT NULL,
    recall_count INTEGER NOT NULL DEFAULT 0,
    UNIQUE (project_id, name)
);
CREATE INDEX idx_entities_project_name ON entities(project_id, name);

CREATE TABLE fact_entity (
    fact_id   TEXT NOT NULL,
    entity_id TEXT NOT NULL,
    PRIMARY KEY (fact_id, entity_id),
    FOREIGN KEY (fact_id)   REFERENCES memory_entries(id),
    FOREIGN KEY (entity_id) REFERENCES entities(id)
);
CREATE INDEX idx_fact_entity_entity ON fact_entity(entity_id);
CREATE INDEX idx_fact_entity_fact   ON fact_entity(fact_id);
```

Backfill iterates `memory_entries.entities` TEXT column, splits on `,`, upserts entities, inserts fact_entity rows. ON CONFLICT DO NOTHING on both UPSERTs — re-runs (interrupted migration recovery) produce identical state.

`memory_entries.entities` TEXT column **preserved**: FTS5 still indexes it; removal deferred to a follow-up BL with one-release-cycle grace period.
